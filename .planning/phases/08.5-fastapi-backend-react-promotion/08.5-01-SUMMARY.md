## What Was Built

### FastAPI App (`src/api/main.py`)
- `FastAPI` app titled "IA West SmartMatch API" v0.1.0
- `CORSMiddleware` allowing `http://localhost:5173` and `http://127.0.0.1:5173`
- 3 routers mounted at `/api/data`, `/api/matching`, `/api/outreach`
- Health check endpoint at `GET /api/health`

### Data Router (`src/api/routers/data.py`)
- `GET /api/data/specialists` — wraps `load_specialists()`
- `GET /api/data/events` — wraps `load_cpp_events()`
- `GET /api/data/pipeline` — wraps `load_pipeline_data()`
- `GET /api/data/calendar` — wraps `load_event_calendar()`
- `GET /api/data/contacts` — wraps `load_poc_contacts()`

### Matching Router (`src/api/routers/matching.py`)
- `POST /api/matching/rank` — calls `rank_speakers_for_event` with full DataFrame context; returns normalized dicts with `name`, `score`, `factor_scores`, `weighted_factor_scores`
- `POST /api/matching/score` — returns single speaker's factor scores within an event ranking

### Outreach Router (`src/api/routers/outreach.py`)
- `POST /api/outreach/email` — builds outreach params via `outreach_bridge`, calls `generate_outreach_email`, returns `{"email": ..., "email_data": ...}`
- `POST /api/outreach/ics` — calls `generate_ics`, returns `{"ics_content": ...}`

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Enhancement] Matching router uses direct engine call instead of data_helpers wrapper**
- **Found during:** Task 2 implementation (pre-existing)
- **Issue:** The plan suggested using `get_top_specialists_for_event` from data_helpers for the rank endpoint, but the existing implementation calls `rank_speakers_for_event` directly with full DataFrames and embeddings for richer factor scores.
- **Impact:** Produces richer normalized output with `factor_scores` and `weighted_factor_scores` — strictly better than the simpler plan spec.
- **Files modified:** src/api/routers/matching.py (already implemented this way)

**2. [Rule 2 - Enhancement] Outreach router reuses _rank_matches internal from matching**
- **Found during:** Task 2 (pre-existing)
- **Issue:** The plan suggested importing `build_outreach_params` and calling data_helpers separately. The implementation instead calls `_rank_matches()` from the matching router to avoid duplicate ranking logic.
- **Impact:** DRY — no code duplication, consistent normalization across rank and email endpoints.

None — all endpoints return real data from the existing business logic modules. Email endpoint falls back to template text when `GEMINI_API_KEY` is not set (documented behavior, not a stub).
